# Eye_Q 공통 공유 문서

## 0. 문서 목적

이 문서는 Eye_Q 시스템을 학생들과 분담 개발할 때 반드시 공통으로 맞춰야 하는 기준을 정리한 문서입니다.

Eye_Q는 여러 명이 각자 따로 프로그램을 만드는 프로젝트가 아닙니다.
하나의 시스템을 여러 모듈로 나누어 개발하는 프로젝트입니다.

따라서 모든 담당자는 아래 내용을 먼저 이해하고, 같은 기준으로 개발해야 합니다.

---

## 1. Eye_Q 시스템 전체 목표

Eye_Q 시스템의 목표는 다음과 같습니다.

```text
컨베이어 위 제품을 카메라로 인식하고,
제품의 위치와 각도를 계산한 뒤,
컨베이어를 정지시키고,
레이저 마킹기에 좌표와 문구를 전달하여,
제품별로 순차 마킹하고,
검사 결과와 마킹 결과를 DB에 저장하는 시스템
```

---

## 2. 전체 공정 흐름

```text
1. 컨베이어 구동
↓
2. 카메라 화면에서 제품 검출
↓
3. 제품이 감지되면 인버터에 정지 명령 전송
↓
4. 컨베이어 정지 후 영상 안정화 대기
↓
5. OpenCV로 제품 경계 검출
↓
6. 중심점 계산
↓
7. 제품 각도 계산
↓
8. 마킹 위치 좌표 보정
↓
9. 레이저 마킹기에 마킹 명령 전송
↓
10. 1번 제품부터 N번 제품까지 순차 마킹
↓
11. 검사 결과와 마킹 결과 저장
↓
12. 다음 사이클 진행
```

---

## 3. 시스템 구성

Eye_Q 시스템은 다음 4개 주요 프로그램으로 구성합니다.

| 구분 | 기술 | 역할 |
|---|---|---|
| 현장 제어 프로그램 | C# WPF | 카메라, 비전, 장비 제어, 자동운전 |
| 서버 프로그램 | ASP.NET Core Web API | DB 저장, API 제공, 인증 처리 |
| 웹 대시보드 | Vue 3 + Vite | 생산 현황, 검사 이력, 알람 조회 |
| 데이터베이스 | PostgreSQL | 검사 결과, 마킹 결과, 알람, 설정 저장 |

---

## 4. WPF와 Vue의 역할 구분

### 4-1. WPF 역할

WPF는 현장 PC에서 실행되는 제어 프로그램입니다.

WPF가 담당하는 기능은 다음과 같습니다.

```text
- 카메라 연결
- 실시간 영상 표시
- OpenCV 검사
- 제품 중심점 계산
- 제품 각도 계산
- 컨베이어 인버터 제어
- 레이저 마킹기 제어
- 자동운전 시퀀스 제어
- 현장 알람 표시
- 서버 API로 결과 전송
```

### 4-2. Vue 역할

Vue는 웹 브라우저에서 접속하는 관리자 화면입니다.

Vue가 담당하는 기능은 다음과 같습니다.

```text
- 생산 수량 조회
- OK / NG 수량 조회
- 검사 이력 조회
- 마킹 이력 조회
- 알람 이력 조회
- 장비 상태 조회
- 제품 설정 조회 및 수정
- 레시피 설정 조회 및 수정
```

### 4-3. 중요한 원칙

```text
WPF = 장비 제어
Vue = 조회 및 관리
ASP.NET Core = 데이터 중계 및 저장
PostgreSQL = 데이터 저장
```

Vue에서 직접 카메라, COM 포트, RS485, 마킹기를 제어하지 않습니다.

## 8-1. DetectedObject

카메라와 OpenCV가 검출한 제품 1개의 정보입니다.

```csharp
public class DetectedObject
{
    public int ObjectNo { get; set; }
    public double CenterXPixel { get; set; }
    public double CenterYPixel { get; set; }
    public double WidthPixel { get; set; }
    public double HeightPixel { get; set; }
    public double AngleDegree { get; set; }
    public bool IsHorizontalOk { get; set; }
    public double Confidence { get; set; }
}
```

### 필드 설명

| 필드명 | 설명 |
|---|---|
| ObjectNo | 화면 내 검출된 제품 번호 |
| CenterXPixel | 제품 중심 X 좌표, 픽셀 단위 |
| CenterYPixel | 제품 중심 Y 좌표, 픽셀 단위 |
| WidthPixel | 제품 가로 길이, 픽셀 단위 |
| HeightPixel | 제품 세로 길이, 픽셀 단위 |
| AngleDegree | 제품 회전 각도 |
| IsHorizontalOk | 수평 허용 범위 안에 있는지 여부 |
| Confidence | 검출 신뢰도 또는 점수 |

## 12. 마킹 순서 기준

화면에서 여러 제품이 검출될 경우 순서를 정해야 합니다.

초기 기준은 다음과 같습니다.

```text
1차 정렬: CenterYPixel 오름차순
2차 정렬: CenterXPixel 오름차순
```

즉, 화면 기준으로 위쪽 줄부터 왼쪽에서 오른쪽 순서로 마킹합니다.

## 13. 좌표 보정 공통 기준

OpenCV가 계산한 좌표는 픽셀 좌표입니다.
레이저 마킹기는 실제 좌표를 사용합니다.

따라서 아래 변환 과정이 필요합니다.

```text
카메라 픽셀 좌표
↓
ROI 기준 좌표
↓
픽셀-mm 변환
↓
카메라와 마킹기 원점 차이 보정
↓
실제 마킹 좌표
```

공통 보정값은 다음과 같습니다.

| 보정값 | 설명 |
|---|---|
| PixelToMmScaleX | X축 픽셀을 mm로 변환하는 비율 |
| PixelToMmScaleY | Y축 픽셀을 mm로 변환하는 비율 |
| OffsetX | 카메라 기준점과 마킹기 기준점의 X 차이 |
| OffsetY | 카메라 기준점과 마킹기 기준점의 Y 차이 |
| RotationOffset | 카메라 설치 각도 보정값 |
| MarkingOriginX | 마킹기 X 원점 |
| MarkingOriginY | 마킹기 Y 원점 |

## 20. 각 담당자가 지켜야 할 원칙

### 20-1. PM / 시스템 설계 담당

```text
- 요구사항 변경을 기록한다.
- 누락 기능을 체크한다.
- 각 담당자의 산출물을 연결한다.
- 통합 테스트 항목을 관리한다.
```

### 20-2. WPF UI 담당

```text
- 화면은 기능 호출만 담당한다.
- OpenCV, Modbus, Laser 로직을 화면 코드에 직접 넣지 않는다.
- 버튼 클릭 시 Service를 호출하는 구조로 만든다.
```

### 20-3. OpenCV 담당

```text
- 검출 결과는 반드시 DetectedObject로 반환한다.
- 중심점, 각도, 크기를 같은 단위로 정리한다.
- 화면에 표시하는 값과 서버에 저장하는 값이 일치해야 한다.
```

### 20-4. 자동운전 담당

```text
- MachineState 기준으로 상태를 관리한다.
- 각 장비 기능은 인터페이스로 호출한다.
- 예외 발생 시 Error 또는 EmergencyStop으로 이동시킨다.
```

### 20-5. Modbus 담당

```text
- 처음에는 FakeConveyorService를 만든다.
- 실제 장비 연결은 나중에 ModbusConveyorService에서 처리한다.
- RUN, STOP, 상태 읽기는 반드시 분리한다.
```

### 20-6. Laser 담당

```text
- 처음에는 FakeLaserMarker를 만든다.
- 실제 장비 방식이 확정되면 Tcp, Serial, SDK 중 하나로 구현한다.
- 마킹 성공 / 실패 결과를 반드시 반환한다.
```

### 20-7. Backend / DB 담당

```text
- API 주소를 문서와 맞춘다.
- DB 테이블명과 DTO 이름을 공통 규칙에 맞춘다.
- 저장 실패 시 오류 응답을 명확히 반환한다.
```

### 20-8. Vue / 테스트 담당

```text
- Vue는 장비를 직접 제어하지 않는다.
- API를 통해 조회하고 관리한다.
- 테스트 체크리스트를 계속 갱신한다.
```

## 24. 최종 통합 기준

```text
WPF 자동운전 시작
↓
Fake 또는 실제 컨베이어 구동
↓
카메라 영상에서 제품 검출
↓
컨베이어 정지
↓
중심점과 각도 계산
↓
마킹 좌표 변환
↓
Fake 또는 실제 레이저 마킹
↓
검사 결과 API 저장
↓
마킹 결과 API 저장
↓
Vue 대시보드에서 결과 조회
```

이 흐름이 완성되어야 Eye_Q 1차 통합이 완료된 것으로 봅니다.
